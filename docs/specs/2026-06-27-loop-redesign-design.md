# 서울 라스트 — 루프 학습 재설계 (설계 문서)

작성일: 2026-06-27
프로젝트: `D:/Unity/MySeoul` (Unity 6000.3.10f1)
성격: 기존 구조를 **갈아엎지 않고** 핵심 재미를 살리는 재설계. 팀 작업물(학교 배경·UI·이벤트·가방·스탯·데이터 파이프라인) 최대 재활용.

## 1. 핵심 재미 (한 줄)
**불합리하게 죽는 날들을 반복하며, 언제 무엇이 오는지·탈출 경로가 무엇인지 "학습"하고, 좁은 가방에 대비책을 챙겨 결국 한 판에 탈출한다.**

- 죽음 = 벌이 아니라 **단서 획득**.
- 학습이 곧 플레이. "아, 이렇게 하면 되는구나"가 도파민.

## 2. 루프 정의 (무엇이 남고 무엇이 리셋되나)
- **리셋**: 아이템, 가방, 진행 조건, 날짜 → 죽으면 1일차로 초기화.
- **유지**: 플레이어의 **지식** = **일기장(누적)**. "3일에 화재", "무전기는 과학실에서 고친다", "탈출엔 연료 필요" 등이 일기에 쌓여 다음 루프에 보임. → 일기장이 *살아있는 공략집.*
- 즉 매 루프 = *경로를 알아가며* 더 멀리, 결국 한 판에 완주.

## 3. 한 판(루프) 흐름
하단 탭 네비게이션 **제거**. 선형:
```
일기장(어제 결과 + 현재 상태 + 배운 것)
  → [확인]
가방 세팅(빈손에서 시작, 가진 것 배치)
  → [하루 시작]
날짜별 이벤트 발생 (그날의 풀에서 1개 랜덤)
  → 선택지 제시: 아이템 보유 시 추가 옵션 / 없으면 도박
  → 결과 (아이템 획득·소모, 상태 변동, 진행 조건 갱신)
  → 다음 날 (Day++, 상태 악화)
  → … 반복
조건 충족 시 → 생존(탈출) 이벤트 발생 → 최종 조건 충족 → 엔딩(승리)
스탯 0 → 사망 → 루프 리셋(지식 유지)
```

## 4. 이벤트 — 날짜 기반 풀 (b안)
- **장소 선택 제거.** 이벤트는 **날짜에 묶임.**
- 날짜마다 **좁은 풀(2~3개)** 중 하나가 랜덤 발생. → "어떤 게 뜰지"는 변주, "이 무렵엔 무엇류가 온다"는 학습 가능.
- **전조/단서 제공**: 이벤트나 일기가 다가올 위협·필요 아이템을 흘려줘야 대비(=학습)가 보답받음.
- 데이터: 기존 `EventData`(+ `choices`) 재활용. `region` 대신 **발생 날짜 범위** 필드를 둠.

## 5. 아이템 & 가방
- **빈손 시작 → 이벤트로 획득.** 초반엔 아이템이 없어 손해를 보다, 모은 것으로 버팀(희소성+성장).
- **가방 = 용량 슬롯 (모양 그리드 제거).** 칸 제한이라 다 못 챙김 → "무엇을 우선?" 결정. 팀의 가방 UI는 *슬롯형으로 단순화*해 재활용.
- **아이템 = 길을 여는 도구** (막는 벽 아님). 보유 시 유리한 추가 선택지가 열림.
- **무도구 행동 = 도박**: 성공(보상)/실패(손해)를 50% 내외로. 아이템은 *확률을 좋게 바꾸거나 확정 옵션을 추가*.
- **내구도 = 결정론(보이게)**: `ItemData.durability` 컬럼 재활용.
  - 소모품(붕대·정수제 등) = 1회.
  - 도구(도끼·손전등 등) = N회, 0이면 파손.
  - 도박 선택에 한해 *약간의 추가 파손 확률* 허용(양념). 평상시는 결정론.

## 6. 승리(엔딩) 조건
- **누적형 탈출**: 이전 선택들의 결과가 **진행 조건(플래그)**을 쌓음.
- 전제 조건이 충족되면 **생존(탈출) 이벤트**가 트리거됨.
- 그 이벤트의 **최종 조건까지 충족**하면 → **엔딩(승리).**
- 예시(데모): `무전기 수리` + `연료 확보` + `지도 입수` 3개 플래그 → "옥상 신호" 탈출 이벤트 발생 → 그때 `무전기`가 가방에 있으면 → 구조/탈출 엔딩.
- **경로는 단서로 발견 가능해야 함**(일기·이벤트가 힌트). 학습→경로 완성→탈출.

## 7. 상태이상
- 기존 5종(허기/수분/건강/기운/기분, 주의/위험 2단계, 정상 시 숨김) 유지.
- 일기장에서 텍스트로, 좌측 알약으로 시각화. (집중 위해 추후 2~3개로 축소 검토 가능 — 이번 범위 아님)

## 8. 제거 / 재활용 정리
- **제거**: 하단 탭 네비, 장소 선택(학교도면 이동), 가방 모양 그리드, 위치 기반 랜덤.
- **재활용**: 일기장(→공략집), 가방 UI(→슬롯), 이벤트 데이터/선택지/내구도 컬럼, 상태이상, 스타트/컷씬/상점/게임오버 더미, Data Tools 파이프라인, 학교 배경.

## 9. 배경(설정)
- 학교 유지(팀 작업). 루프의 명분(밤/봉쇄/저주 등)은 *나중에 한 겹* 칠함 — 메커닉 확정이 우선.

## 10. 이번 범위 밖 (나중)
- 컷씬·연출·아트, 상점 실제 구매/화폐, 배경 명분 서사, 기획자 실데이터 대량 연동, 상태이상 축소.

## 11. 성공 기준
- 회색박스로 한 루프를 돌렸을 때: **"다음 판엔 이렇게 해야지"**가 자연히 떠오르면 성공. (학습 욕구가 생기는가)

# 최종 확정 (회의 후, 2026-06-27) — §1~10 덮어씀

## A. 이벤트 — 날짜 기반
- 이벤트는 **발생 날짜(day) 풀**에 묶임. 날짜마다 2~3개 중 랜덤 1개.
- **특정 일자엔 "탐사(장소) 이벤트"**로 등장 — 이전처럼 **필요 아이템 표시(가이드)**. "지금 없어도 다음에 챙겨놔야지" 유도.

## B. 인벤토리 — 그리드 백팩 only (창고 제거)
- **가방 모양 그리드는 유지** (6×5 고정, 아이템마다 모양: 도끼 ㄱ자, 구급상자·무전기 2×2 등). 한정된 그리드 공간이 곧 제약 — 획득품을 **넣을지/버릴지 고민**하게 만드는 게 핵심.
- **창고(별도 보관) 개념만 제거.** 그리드에 들어간 것 = 실제 보유. **빈손 시작 → 이벤트로 획득.**
- 획득 시: 자동 배치 시도 → 자리 없으면 **획득 대기(트레이)** 로. 정비 화면에서 드래그로 그리드에 넣거나 **버리기 존**에 버림. 하루 시작 시 미배치분은 **버려짐**.
- **아이템 = 길을 여는 도구.** 없을 때: **① 반반(50%) 도전 / ② 행동 포기** 둘 다 제공.
- **내구도(결정론·표시)**: 사용 횟수 소진 시 **파괴**(`ItemData.durability`). **분실 이벤트**도 존재.

## C. 상태이상 — 5 → 4, 아이템으로 회복
- 4종: **허기 / 수분 / 건강 / 기운** (주의·위험일 때만 표시). 매일 악화, 하나라도 100이면 사망.
- **아이템 사용으로 회복** (통조림→허기, 물→수분, 구급상자→건강, 각성제→기운).

## D. 엔딩 — A / B / C (D 없음)
- **시작 분기점이 날짜별로 열림**: A≈Day3, B≈Day5, C≈Day7.
- 완성 = 그 엔딩 체인 선택지에서 **정답 3연속**.
- **틀리면 그 엔딩만 무효** (다른 엔딩은 시도 가능). 시차로 A 실패해도 B/C 노림.
- 정답은 **루프 돌며 학습.**

## E. 루프
- 죽으면 **아이템·진행 전부 리셋, 시작 화면부터 (컷씬 스킵).**

## F. 흐름 간소화 (2026-06-27)
- **일기장 화면 제거.** 하루 시작점은 **가방 정비**. 흐름: 시작 → (초회)컷씬 → **가방 정비** → 하루 시작 → 날짜 이벤트 → 결과(확인) → 가방 정비 → … (지난 결과는 이벤트 결과 화면에서 바로 확인). 상태 요약은 가방 정비 화면에 표시.

## G. 이벤트 테이블 연동 — 분기 그래프 (2026-06-27)
- 기획 구글시트 event 테이블이 **분기 그래프 구조**로 변경. Data Tools ③(Event.csv → EventData)가 새 컬럼을 임포트:
  - 기본: EventType, EventHasBranch, EventRegion, EventObjectID, EventDescription(→situation)
  - 분기 A/B/C: 버튼문구 / 결과 / **OpensInventory**(가방 열림) / **NewState**(상태 변경) / **NextEventId**(다음 이벤트)
  - 유발 상태이상(마지막 컬럼, 영어 헤더 비어 위치로 매핑). C 결과 헤더는 시트 오타(EventBranchBResult)라 **위치로 읽음**.
- **GameFlow는 새 시트 EVT 이벤트로 교체.** 온보딩은 **EVT-O001**부터 시작해 NextEventId 그래프로 이동.
- **분기(버튼)가 비면 → 터치(확인)하면 다음 이벤트.** 분기가 있으면 버튼 표시(아이템 부족 시에만 도박/포기 보강).
- **NewState = 컨디션 부여**: 예 "배고픔"이면 허기를 주의 임계치(40)까지만 악화, 이미 그 이상이면 변화 없음. 위험어(굶주림 등)=70. "스탯±N"(예 "기운+15") 토큰도 지원. 불안/중독/공포/화상 등 4스탯 외 상태이상은 현재 무시.
- **OpensInventory** 선택 시: 결과 후 가방 정비 화면으로 이동(아이템 배치).

## H. 코어 루프 — 스테이지 단위 (2026-06-27)
- **날짜 루프 → 스테이지 루프로 전환.** 흐름:
  ```
  스타트 → (초회)컷씬 → [온보딩(스테이지0): EVT-O001부터 NextEventId 그래프] → 휴식
    → 지도(지역 선택) → 스테이지 N(이벤트 5개) → 휴식 → 지도 → … 반복
  ```
- **1스테이지 = 5 이벤트.** 5개째 종료 시 **휴식화면**으로. 휴식에서 가방 정비 + **지도**로 다음 장소 선택.
- **온보딩(스테이지 0)**: EVT-O001부터 시작, 분기 NextEventId 그래프대로 이동. 다음이 없으면(데이터 종단) 휴식으로. 온보딩 중엔 시간경과(상태 악화)/사망/5카운트 없음 — 튜토리얼.
- **실전 스테이지(1+)**: 첫 이벤트는 random(지역+일반), 이벤트마다 상태 악화 + 사망 판정 + 카운트. 5개 후 휴식.
- **NextEventId == "random"**: 온보딩 제외, (선택 지역의 특정지역 + 일반) 풀에서 랜덤.
- **지도**: 특정지역 이벤트의 distinct 지역(예 음악실/과학실)을 버튼으로. 선택 시 그 지역이 스테이지 지역이 됨.
- 죽으면 리셋(ResetRun) → 시작화면 → 온보딩부터 다시(현재. 추후 온보딩 1회성 게이트 검토).

## I. FlowCanvas 씬 배치 (UI 개발자 협업, 2026-06-27)
- FlowCanvas를 **씬에 실제 GameObject로 배치** → UI 개발자가 레이아웃/색/폰트/스프라이트를 에디터에서 직접 편집.
- 메뉴 **NoPainYesGame ▸ Build Flow Canvas In Scene** 로 생성(기존 `GameFlow.BuildUI` 코드를 에디트 모드에서 실행 → 동일 계층, 에셋 폰트 `Assets/Fonts/malgun.ttf` + 빌트인 UISprite로 교체해 직렬화).
- **런타임 동작**: `GameFlow.Awake`가 씬의 FlowCanvas를 발견하면 이름으로 참조 바인딩 + 버튼 onClick만 코드로 연결(`BindUI`). 없으면 기존처럼 코드로 생성(폴백).
- 즉 **모양 = 씬(디자이너), 동작 = GameFlow(코드)** 분리. 동적 콘텐츠(선택지·그리드 아이템·지도 버튼 등)는 런타임에 해당 컨테이너로 생성됨.
- ⚠ 버튼/참조는 **이름·계층 경로**로 바인딩하므로, UI 개발자는 패널/버튼 오브젝트 **이름과 부모-자식 구조를 유지**해야 함(스타일·위치·크기·스프라이트·폰트는 자유). 구조가 크게 바뀌면 메뉴로 재생성(단, 재생성은 수동 스타일 작업을 덮어씀).

## 구현 단계
1. 데이터: `EventData`에 day범위+엔딩 필드, `EventChoice`에 정답/회복/분실. 상태 5→4.
2. `GameFlow` 코어: 날짜 이벤트 루프 + 4스탯 + 도박/포기 + 루프 리셋(컷씬 스킵).
3. 인벤토리: 백팩 only + 내구도 + 공간부족 버리기 + 아이템 사용 회복.
4. 엔딩: A/B/C 시차 + 정답 3연속 + 실패 시 해당 엔딩만 무효.
5. 데모 이벤트(날짜별 + 엔딩 체인) 작성, 빌드·검증.
